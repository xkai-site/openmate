# 虚拟文件系统架构

## 1. 模块定位

这里的“虚拟文件系统”不是对真实文件系统的简单映射，而是 AINative OS 在应用层上的组织模型。

它的目标不是还原传统目录树，而是给用户提供一个足够自然、足够稳定的工作空间抽象：

1. 用户直接面对的对象只有 `Topic` 和 `Node`。
2. `Topic` 表示一个项目、或者一个稳定的问题域边界。
3. `Node` 表示项目内部的具体对象，它既可以承担“容器”角色，也可以承担“可执行单元”角色。
4. 用户只能直接操作叶子 `Node`，也就是当前没有子节点的 Node。

因此，这一层本质上是在回答两个问题：

1. 用户如何组织自己的工作内容。
2. 系统如何把这些内容变成可追踪、可调度、可协作的数据结构。

## 2. 核心语义

### 2.1 Topic

`Topic` 是“项目”而不是“文件夹”。

它的职责是：

1. 提供一个独立的工作边界。
2. 承载该项目下所有 Node 的集合。
3. 作为后续调度、记忆、权限和观测的基本归属单位。
4. 作为项目级业务计量的聚合单位。

因此，`Topic` 不强调层级操作本身，它强调的是“归属”和“上下文边界”。

### 2.2 Node

`Node` 是系统中的统一对象模型。

它没有固定的“文件”或“文件夹”身份，而是根据自己在树中的位置承担当前职责。

在当前设计中，它有两种常见结构状态：

1. 父节点：用于组织其他 Node，承担结构作用。
2. 叶子节点：用于被用户直接操作，承担具体协作与执行作用。

这意味着 `Node` 不是单纯的文件，也不是单纯的文件夹，而是统一抽象后根据结构位置表现出不同职责。

用户通常从根节点直接开始交互，根节点在初始阶段也可以是可操作节点；只有当它真正展开出子节点后，它才转为父节点。

从业务计量角度，`Node` 也是最小计量单元。

### 2.3 叶子节点约束

用户只能直接操作叶子 Node，这条约束是整个系统的重要设计哲学。

这样做的好处是：

1. 交互入口稳定，不会在容器节点上混入执行语义。
2. 调度入口清晰，系统只需要面向叶子 Node 直接调度与执行。
3. 结构节点可以专注于组织、归纳和聚合，而不是承担具体执行。

## 3. 结构模型

在当前设计中，一个 `Topic` 内部采用以 `Node` 为元素的树结构。

这里的树结构主要服务于用户心智模型。

也就是说，用户看到的是一棵项目树，能够自然地理解、浏览和操作其中的内容；当前阶段的虚拟文件系统也只需要围绕这棵树组织对象。

树关系由以下字段表达：

1. `parent_id`：指向父节点。
2. `children_ids`：指向子节点列表。

当前阶段，这一层只定义和维护树结构本身，不额外展开其他关系模型。

## 4. 数据结构

### 4.1 Topic

`Topic` 建议至少包含以下字段：

1. `id`
2. `name`
3. `root_node_id`
4. `metadata`
5. `description`
6. `tags`
7. `created_at`
8. `updated_at`

说明：

1. `root_node_id` 用于显式确定该 Topic 的树根。
2. `metadata` 是项目级元数据，它只保存非常必要的信息，并且属于每次执行的必读内容。
3. `description` 和 `tags` 不是必须，但对检索、分类和调度有帮助。

### 4.2 Node

结合当前讨论，`Node` 的核心字段建议定义如下。

必选字段：

1. `id`
2. `topic_id`
3. `name`
4. `description: str | null`
5. `parent_id: str | null`
6. `children_ids: list[str]`
7. `session: list[session_id]`
8. `memory: KV | null`
9. `input: KV`
10. `output: KV`
11. `progress: list[str]`
12. `status: string`
13. `created_at`
14. `updated_at`

说明：

`topic_id` 建议直接保存在 Node 上，而不是完全依赖树关系推导。

推荐补充字段：

1. `version`

这些补充字段的作用如下：

1. `version`
作用：支持并发写控制，后续可以用于乐观锁。
当前 Go CLI 已通过 `node update --expected-version` 使用该字段做乐观并发校验。

### 4.3 Node 字段语义说明

1. `session`
表示与该 Node 绑定的对话上下文，按时间顺序存储引用。这里更适合存 `session_id` 列表，而不是把完整对话直接塞进 Node。
当前 Go 实现已把 Session 正文独立落到 SQLite：`Session` 负责元信息，`SessionEvent` 负责顺序追加的事件流；`Node.session` 继续只保存 `session_id` 引用。
Session 与 SessionEvent 的当前生效字段、状态机与兼容策略，不在本文重复维护，统一以 [Session与SessionEvent字段收敛.md](D:/XuKai/Project/vos/architecture/虚拟文件系统/Session与SessionEvent字段收敛.md) 为准。
2. `description`
表示该 Node 的稳定摘要，用于补充 `name` 无法表达的目标、范围或阶段信息。它应保持轻量，适合被调度、检索与能力层装配复用，而不是承载完整过程数据。
3. `memory`
表示该 Node 的长期记忆。它是核心属性，但允许为空；它主要承担写入和汇总来源的职责，而不一定是执行时的直接读取对象。
4. `input`
表示用户上传或外部输入的逻辑索引。其实际内容建议落在文件系统或对象存储中，这里只保留索引信息。
5. `output`
表示 Node 的结果索引。和 `input` 一样，建议保存引用而非大块原始内容。
6. `progress`
MVP 阶段可以先用 `list[str]`，表达进展日志；后续如果需要更强检索和审计能力，可以升级为结构化记录。
7. `status`
表示 Node 当前状态，应统一约束为有限状态集合，而不是完全自由字符串。

## 5. 记忆机制

这部分是虚拟文件系统设计中的核心理念之一。

这里的记忆机制不是简单地“每个节点都读自己的 Memory”，而是带有明确缓存层次和读写分离哲学的设计。

### 5.1 基本判断

从实际使用来看，真正需要稳定存储记忆的通常只有两类节点：

1. 叶子 Node 本身。
2. 叶子 Node 的父节点。

更高层节点理论上也可能存在记忆，但通常不是常态。少数情况下会因为节点上浮、下移或结构调整，导致更高层节点暂时持有记忆。

### 5.2 多级缓存结构

执行时的上下文读取分成四层：

1. `Topic.metadata`
项目级元数据，只存储非常必要的信息，每次必读。
2. `session`
当前 Node 的对话上下文，每次都会注入。
当前阶段会话正文不再跟随 `.vos_state.json` 一起整体重写，而是落在独立 Session SQLite 中，由 VFS 在执行时按 `session_id` 读取。
工具调用相关的跨模块事件契约，统一引用 `architecture/sharedInfo/工具调用-SessionEvent契约.md`。
3. `parent memory`
父节点记忆，每次都会注入。它承担共享记忆层的作用。
4. `record table`
只保存 Node 的索引与名称，类似 skill 一样注入，但不一定会被读取，按需获取。

这四层中，真正始终稳定参与执行的，是 `Topic.metadata + 当前 session + 父节点 memory`；记录表属于辅助层，主要用于低成本提供结构感知。

### 5.3 读写分离

这套记忆机制的哲学在于读写分离。

写入时：

1. 先写当前叶子 Node 自己的 `memory`。
2. 再通过工程过程更新其父节点的 `memory`。

读取时：

1. 不直接依赖当前 Node 自己的 `memory` 作为主要注入源。
2. 主要读取父节点的 `memory`，因为父节点已经聚合了下一层节点的记忆，能够被多个叶子共享。

这样设计的结果是，执行层不需要在每次运行时反复拼接多个细粒度记忆块，压力会小很多。

### 5.4 父节点记忆的作用

父节点 `memory` 的定位不是简单备份，而是子节点记忆的聚合层。

它的作用主要有三点：

1. 为同一父节点下的多个叶子 Node 提供共享上下文。
2. 避免执行时逐个回收、拼装多个子节点的记忆。
3. 让“读路径”稳定落在较少的上下文源上。

因此，叶子 Node 的 `memory` 更像写入源，而父节点 `memory` 更像读取层。

父节点 `memory` 是共享冗余缓存，不要求与当前直接子节点集合严格一一对应。
当前 Go + JSON Store 的实现会把这层聚合缓存写入父节点 `memory._child_memory_cache` 保留键，存储子节点 `memory` 快照；纯结构变化本身不触发该缓存重算。

### 5.5 记录表

记录表只存 Node 的索引与名称。

它的定位不是长期记忆本体，而是轻量辅助信息：

1. 让模型知道当前项目中有哪些 Node。
2. 提供类似 skill 清单的结构感知。
3. 在不展开完整内容的情况下提供按需检索入口。

当前阶段，记录表可以被视为由树结构派生出的辅助索引，不需要把它设计成复杂的核心对象。

进一步来说，记录表挂在独立索引表中，位于 Topic 之外，用于提供跨 Node / Topic 的快速索引能力。

当前阶段只需要知道它存在，不把它纳入当前虚拟文件系统主数据结构，也不进入当前阶段的开发重点。

### 5.6 更新触发时机

父节点记忆的更新不应在每次对话后都发生。

当前设计中，更新父节点记忆的触发时机只发生在子节点记忆变更时，例如：

1. 上下文过长，需要压缩总结。
2. 用户明确确认某项信息应进入长期记忆。
3. 工程流程判断该节点记忆已经发生有效变化。

也就是说，`session` 的频繁变化不等于 `memory` 的频繁变化；只有当记忆层发生变化时，才触发父节点聚合更新。

同时，纯结构变化本身不自动触发记忆重算。

具体来说：

1. `子 -> 父` 时，不修改原父节点的 `memory`。
2. `父 -> 子` 时，不做原父节点数据向下迁移，后续数据由新的子节点体系自己产生。
3. 结构调整本身不天然等于记忆变化。

### 5.7 删除策略

删除叶子 Node 时，核心问题不是结构删除本身，而是是否处理记忆。

当前策略可以定义为：

1. 删除节点时先询问“是否需要删除记忆”。
2. 一般情况下不需要额外操作，因为父节点已经存储了冗余记忆。
3. 如果用户明确要求删除记忆，则调用模型对比父节点记忆并执行清理。

这意味着删除结构与删除记忆是两个分离动作。

## 6. 关键约束

这一层最重要的不是字段多少，而是约束必须稳定。

建议先冻结以下约束：

1. 一个 Node 只能属于一个 Topic。
2. 一个 Topic 内必须存在且仅存在一个根 Node。
3. `parent_id is null` 只允许出现在根 Node 上。
4. `children_ids` 与 `parent_id` 必须保持一致。
5. 树中不能出现环。
6. 用户只能直接操作叶子 Node。
7. 非叶子 Node 不承担具体执行，只承担组织和归纳职责。
8. `session` 必须保持顺序追加语义。
9. `memory` 采用读写分离原则：写当前节点，读父节点。
10. 父节点 `memory` 只在子节点记忆发生变化时更新。
11. 结构变化本身不自动触发记忆重算。
12. 父节点 `memory` 是共享冗余缓存，不要求与当前直接子节点严格一一对应。
13. 删除 Node 与删除记忆是分离动作。
14. `input/output` 以文件引用为主，不建议直接把大对象嵌入 Node。

## 7. 技术栈选择

当前阶段建议明确以下技术方向：

1. 开发语言使用 Go。
2. 数据模型使用 Go 强类型结构体，校验逻辑由领域层与服务层显式维护。
3. 时间字段统一使用标准时间类型，而不是裸字符串。
4. 状态类字段应尽量收敛为枚举或受控字面量集合。

这一选择的好处是：

1. 类型约束和模块边界更稳定，适合作为独立 CLI 工具长期维护。
2. 单二进制交付更直接，后续与其他模块通过 CLI/JSON 对接成本更低。
3. 后续无论底层存 SQLite、PostgreSQL 还是文件索引，领域模型都能保持稳定。

## 8. 存储原则

这一层建议遵循“轻节点、重引用”的思路。

也就是说：

1. `Node` 主要保存结构信息、状态信息和索引信息。
2. 对话正文、上传文件、生成产物应尽量存放在外部文件层。
3. `Node` 中保存的是对这些内容的逻辑引用，而不是把所有内容内联进去。
4. `memory` 是高价值摘要信息，不追求和 `session` 等量齐观。
5. 父节点 `memory` 允许保存对子节点信息的冗余聚合，以换取读取效率。
6. 原始模型调用计量来自 Agent 池，而 Node / Topic 承接业务计量与聚合。

这样做的好处是：

1. Node 本身更稳定，不会因为内容膨胀而变成高频大对象。
2. 后续更容易做检索、缓存、归档和迁移。
3. 对调度层和能力层来说，Node 可以始终保持为一个轻量控制平面对象。
4. 执行层读取路径更短，记忆注入成本更可控。
5. 计量分层更清晰，资源层与业务层不会互相污染。

## 9. 与其他模块的边界

虚拟文件系统模块只负责：

1. 定义 `Topic` 和 `Node` 的组织模型。
2. 管理树结构、节点状态和内容索引。
3. 承接 Node / Topic 级别的业务计量聚合。
4. 为后续模块提供稳定的对象边界。

它不负责：

1. 决定调度优先级。
2. 管理 Agent 生命周期。
3. 执行能力注入和模型调用。
4. 生成底层模型调用的原始计量。

换句话说，这一层是“结构与状态层”，不是“执行层”。

## 10. MVP 范围

文件系统模块的 MVP 建议控制在以下范围：

1. Topic 的基础创建与查询。
2. Node 的基础创建、移动、删除与查询。
3. 基于 `parent_id / children_ids` 的树结构维护。
4. 叶子 Node 的可操作校验。
5. `session / memory / input / output / progress / status` 的基础挂载能力。
6. 节点内容与文件引用的分离。
7. 基于“子写父读”的记忆更新流程。
8. Topic `metadata` 的基础承载方式。

当前阶段先不要过早进入以下内容：

1. 复杂权限系统。
2. 跨 Topic 协同机制。
3. 向量检索和高级记忆策略。
4. 独立索引表 `record table` 的具体开发。
5. 精细化调度接口设计。

## 11. 当前结论

这部分现在可以这样定性：

1. `Topic` 是项目边界，不是传统文件夹。
2. `Node` 是项目中的统一对象，不是单纯文件。
3. 叶子 Node 才是用户直接操作的对象。
4. 树结构主要服务于用户心智模型，而不是试图穷尽协作语义。
5. 记忆机制采用多级缓存与读写分离：子节点写自己的记忆，执行时主要读父节点记忆。
6. Topic 需要提供必读的 `metadata`，记录表作为独立外部索引按需注入。
7. Agent 池负责原始计量，Node 是业务计量最小单元，Topic 是业务计量聚合单元。
8. 技术实现上使用 Go，先把数据结构、CLI 边界和核心约束定义稳定，再继续往调度与执行层推进。
