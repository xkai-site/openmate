# 调度队列架构

## 1. 模块目标

调度层负责把“待执行工作”公平且高效地分配给 Agent 执行资源，兼顾用户交互响应、吞吐与成本。

核心目标：

1. 一级按主题调度，解决“用户当前最关心什么”。
2. 二级按主题内工作流调度，解决“同主题内先做什么”。
3. 与 Agent 解耦，支持动态并发与抢占。
4. 可解释调度决策，便于调优和排障。

## 1.1 技术栈与对接边界

1. 调度层确认采用 Go 技术栈推进，与 `vos/pool` 保持同语言、同仓库、同根模块。
2. 模块间通信统一通过 CLI + JSON，不因同语言而直接复用其他模块的内部包。
3. 调度层对接 VFS 时，应通过 `vos` CLI shell-out + JSON。
4. 调度层对接 Pool 时，应通过 `openmate-pool` CLI shell-out + JSON。
5. 调度层对接 Python Agent 时，边界定义为独立 worker CLI + JSON 契约；在该契约冻结前，不在本文档里假定已有稳定命令。
6. `cmd/openmate-schedule` 是调度模块的唯一对外命令入口，`internal/schedule` 只服务调度模块自身实现。

## 2. 设计结论

采用双队列：

1. 一级队列（TopicQueue）：主题级多级反馈队列（MLFQ）。
2. 二级队列（NodeQueue）：主题内部的可执行叶子 Node 优先队列。

调度原则：

1. 调度与执行分离。
2. Agent 是执行槽位，不与队列强绑定。
3. 一级队列使用操作系统最纯粹的MLFQ机制，管理的是Topic。
4. 二级队列由 Topic 内叶子 Node 的状态与工作流关系共同决定，解决同主题任务下先后顺序。
5. Agent affinity 只作为派发优化项，不作为队列切分依据。
6. 同级轮转防止饿死，时间片耗尽降级，周期性提级。

## 3. 一级队列（主题级）

### 3.1 队列分层

建议四级：

1. `L0`：最高优先，强交互主题。
2. `L1`：活跃主题。
3. `L2`：一般主题。
4. `L3`：低频主题。

### 3.2 进入与迁移规则

1. 新激活主题默认进入 `L0`。
2. 主题用完时间片且仍有待执行任务，降一级。
3. 定时全局提级，避免长期饥饿。
4. 发生新交互事件时按热度可直接提至最高级。

### 3.3 热度计算（建议）

这里的 `score` 只回答一个问题：用户当前注意力主要在哪个 Topic 上。

它不是业务价值评分，也不是让系统猜“哪个任务更重要”。

MVP 阶段建议先采用容易解释的规则：

1. 用户当前正在交互的 Topic 直接提升到高优先级。
2. 最近有用户消息、手动操作的 Topic 提级。
3. 长时间无交互的 Topic 按衰减规则降级。
4. 同等热度下，最近已经拿到较多执行份额的 Topic 可加轻微惩罚，避免独占。

如果后续需要量化实现，再落成启发式分数：

```text
score = interaction_heat
      - idle_decay
      - recent_service_penalty
```

其中：

1. `interaction_heat` 由 `recent_user_messages / recent_manual_actions / explicit_focus_flag` 构成。
2. `idle_decay` 用于让长时间无交互主题自然回落。
3. `recent_service_penalty` 用于抑制热点主题长期占满全部 Agent。

最终按阈值映射到 `L0~L3`。

## 4. 二级队列（主题内工作流）

二级队列位于每个 Topic 内部，它维护的是该 Topic 中可执行叶子 Node 的执行优先级。

它的职责是：当某个 Topic 已经从一级队列获得执行份额后，从该 Topic 的叶子 Node 中挑出最值得先执行的一批 Node。这里的队列是优先队列，优先级计划由人/Agent确定，可以边做边调整。

换句话说：

1. 一级队列解决“现在哪个 Topic 该获得系统服务”。
2. 二级队列解决“这个 Topic 内下一步在哪个 Node 上工作”。
3. Agent 不拥有自己的专属任务队列，而是被派发到具体 Node 上工作。

### 4.1 核心语义

1. 二级队列归属于 `Topic`，而不是归属于 `Agent`；每个 Topic 独立维护自己的局部工作队列。
2. 二级队列的主对象是同一 Topic 下所有“未终态”的叶子 `Node`，因为叶子 Node 才是用户直接操作、也是 Agent 直接工作的对象。
3. 二级层采用 continuation-first 策略：默认优先继续当前焦点 Node，而不是每次重新审视整个队列。
4. `Topic` 需要额外维护一个 `current_node_id`，表示“当前默认应该接着做谁”。
5. PriorityAgent 在新 Node 入队、阶段脏化、当前焦点失效或用户显式干预时重算 Node priority；`current_node_id` 的具体切换由调度器结合 active layer 完成。
6. `Node` 是长期状态与产物的承载体，负责统一沉淀 `session / input / output / progress / status / memory`。
7. 调度层不再引入额外任务对象；Node 自身就是“接着做”的对象。
8. 同一个 Node 在生命周期中可以被多次调度；每次执行都继续在该 Node 上追加状态、对话和产物。
9. 同一 Node 同一时刻最多只能被一个 Agent 处理。

### 4.2 队列、优先级层与运行集

二级队列在这里不应再理解成先进先出的消费队列。它更接近 Topic 内部的一套运行时控制结构：队列中保留所有未终态叶子 Node，真正决定“现在能跑谁”的，是这些 Node 当前被赋予的 priority 所形成的最高优先级层。

二级层的语义重点不是公平分发，而是工作流阶段约束。低优先级 Node 被高优先级 Node 长时间压住，在这里是正常现象，而不是调度缺陷。

建议 Topic 至少维护以下运行时状态：

```ts
interface TopicRuntimeState {
  topicId: string;
  active_priority?: string;
  current_node_id?: string;
  running_node_ids: string[];
  last_worked_node_id?: string;
  last_worked_at?: string;
  switch_count: number;
}
```

其中：

1. 队列元素仍是同一 Topic 下所有未终态的叶子 `Node`。
2. 每个 Node 都有当前调度 `priority`，可存放在调度索引中，不要求污染 Node 的核心业务结构。
3. `active_priority` 表示当前 Topic 内唯一允许被执行的最高优先级层。
4. `current_node_id` 表示当前主焦点 Node，它是 active layer 内的默认续做对象，而不是唯一运行对象。
5. `running_node_ids` 表示当前真正拿到 Agent 并正在执行的 Node 集合。
6. `last_worked_node_id` 表示最近一个被真实推进过的 Node。
7. `last_worked_at` 表示最近一次真实推进该 Topic 的时间。
8. `switch_count` 表示主焦点切换次数。

这几个字段的职责边界如下：

1. 队列回答“这个 Topic 里还有哪些 Node 处于管理范围内”。
2. `active_priority` 回答“当前哪一层可以动”。
3. `current_node_id` 回答“在可动那一层里，默认优先续做谁”。
4. `running_node_ids` 回答“现在实际上有哪些 Node 正在干活”。
5. `last_worked_node_id / last_worked_at` 回答“最近真正落地推进的是谁、发生在什么时候”。
6. `switch_count` 回答“该 Topic 是否频繁切换主焦点”。

由此会自然形成几种状态：

1. `active_priority` 为空：该 Topic 当前没有可调度的未终态叶子 Node。
2. `current_node_id` 在 `running_node_ids` 中：主焦点已经落地，处于稳定推进状态。
3. `current_node_id` 不在 `running_node_ids` 中，但仍属于 `active_priority`：焦点已经选定，但还没有真正拿到 Agent 落地执行。
4. `current_node_id != last_worked_node_id` 已不再足以判断“切换待落地”，因为同一优先级层允许并发；真正可用的判定条件是 `current_node_id not in running_node_ids`。

### 4.3 调度决策（建议）

并发升级后，二级层不再是“单焦点串行队列”，而是“严格优先级层 + 同层并发”。

核心规则：

1. 先确定 `active_priority`。
2. 只有 `priority == active_priority` 的 Node 允许执行。
3. 同一优先级层可被多个 Agent 并发消费。
4. 同一个 Node 同时最多只能被一个 Agent 处理。
5. 低优先级 Node 即使等待很久、即使系统还有空闲 Agent，也不能偷跑。

建议调度过程如下：

```text
1) 若 Topic 的优先级索引被标记为 dirty，则调用一次 PriorityAgent 重算未终态叶子 Node 的 priority
2) 计算 active_priority = 当前所有未终态叶子 Node 中的最高 priority
3) 取出 active_candidates =
   priority == active_priority
   and status not terminal
   and not blocked
   and not in retry_cooldown
4) 若 active_candidates 为空：
   说明当前最高优先级层整体阻塞，Topic 进入 stall，不能下沉到更低优先级层
5) 若 current_node_id 不在 active_candidates 中：
   使用工程规则重置主焦点
6) 若 current_node_id 可派发且未在 running_node_ids 中：
   优先派发 current_node_id
7) 若还有空闲 Agent：
   继续派发 active_candidates 中其余未运行 Node
8) 派发成功后，把对应 nodeId 加入 running_node_ids
9) Agent 真正开始工作或写入首个有效进展时，更新 last_worked_node_id / last_worked_at
10) 任一 active layer Node 完成、失败、阻塞、取消、恢复后，按需重算 priority 与 active_priority
```

这里有两个容易误解的点：

1. `current_node_id` 仍然保留，但它只负责“默认先续做谁”，不再代表“Topic 内只能跑这一个 Node”。
2. 当 active layer 内有多个 Node 时，系统可以同时派发多个 Agent；但这些 Agent 只能来自同一优先级层。

`current_node_id` 的选择不必再交给 PriorityAgent。建议使用纯工程规则，保持职责收敛：

1. 如果原 `current_node_id` 仍在 active layer 且可继续，直接保留。
2. 否则优先复用仍可继续的 `last_worked_node_id`，减少切换损耗。
3. 再否则，从 active layer 中按稳定顺序选一个作为新焦点，例如最早进入该层或等待最久的 Node。

一个典型例子：

1. 某 Topic 下有 5 个未终态 Node。
2. 其中只有 A、B 属于当前最高优先级层。
3. 系统此时有 5 个空闲 Agent。
4. 结果只能派发 A、B。
5. 其余 3 个 Agent 要么空闲，要么去服务别的 Topic，但不能提前处理更低优先级的 C、D、E。

### 4.4 工程侧 AOP 维护

这些运行时字段应由系统横切维护，而不是让 Agent 自己记。

建议至少在以下时机自动更新：

1. PriorityAgent 回写新的 priority 结果后，系统自动重算 `active_priority`。
2. `active_priority` 变化并导致主焦点失效时，系统自动重选 `current_node_id`。
3. `current_node_id` 从 A 切到 B 时，系统自动执行 `switch_count += 1`。
4. 派发成功后，系统自动把对应 Node 加入 `running_node_ids`。
5. 执行结束、失败、取消、超时、失租后，系统自动把对应 Node 从 `running_node_ids` 移除。
6. Agent 真正开始工作或产出首个有效进展时，系统自动更新 `last_worked_node_id / last_worked_at`。
7. 当前主焦点进入终态、阻塞态、重试冷却或跌出 active layer 时，系统自动失效化 `current_node_id` 并触发重选。
8. Topic 长时间无有效推进时，系统自动打点 stall 指标，而不是偷偷放行低优先级层。

需要特别补充的是：

1. `current_node_id` 是控制面字段，代表“系统决定下一步主看谁”。
2. `running_node_ids` 是执行面字段，代表“实际有哪些 Node 已落地运行”。
3. `last_worked_node_id` 是事实字段，代表“最近一次真正干活的是谁”。

因此完全可能出现下面这种状态：

1. PriorityAgent 重算后，B 被提升到最高优先级层。
2. 系统立即把 `current_node_id = B`。
3. 但此时还没有空闲 Agent，或者 B 还未真正开始执行。
4. `running_node_ids` 里仍没有 B。
5. `last_worked_node_id` 甚至还停留在 A。

这不是脏数据，而是一个合法中间态：切换决定已经生效，但执行尚未落地。

也正因为如此，在并发版本里：

1. `current_node_id != last_worked_node_id` 不能直接等价于“切换待落地”。
2. 更准确的判断方式是：`current_node_id` 是否已经进入 `running_node_ids`。

### 4.5 优先级 Agent 的职责

PriorityAgent 的职责需要收窄：它负责给 Node 赋予或重算 priority，不负责直接返回 `next_node_id`。

它不是常驻调度器，也不接管派发循环。它的输出应该是“层级判断”，而不是“把哪个 Agent 派给谁”。

建议接入时机：

1. 新的叶子 Node 入队时。
2. 当前 active layer 有 Node 完成或进入终态时。
3. 某个 Node 进入阻塞、恢复、重试冷却或解除冷却时。
4. 用户追加新指令，导致工作阶段可能变化时。
5. 当前 Topic 长时间无有效进展，需要重新审视阶段优先级时。

PriorityAgent 读取的是全队列轻量快照。

建议输入至少包含：

1. `nodeId`
2. `name`
3. `description`
4. `status`
5. `progress`
6. `last_worked_at`
7. `isCurrent`
8. `isRunning`

必要时可额外补充少量人工 hint，但不建议把完整 session 全量塞给它。

建议输出至少包含：

 `reason_tag`

系统接到结果后的动作是：

1. 更新对应 Node 的调度 priority。
2. 重算 `active_priority`。
3. 若当前主焦点已不在 active layer，重选 `current_node_id`。
4. 绝不让 PriorityAgent 直接操作 `running_node_ids` 或 Agent 分配。

这样分工的好处是：

1. “这个阶段先做什么”交给 Agent 判断。
2. “当前有没有空闲 Agent、能派几个、谁正在跑”交给工程调度器处理。
3. 优先级判断与执行派发解耦，后续更容易做缓存、观测和回放。

### 4.6 调度层看到的执行语义

这样设计后，调度层与执行层的边界非常清楚：

1. 调度层只负责决定“哪个 Topic 当前得服务、这个 Topic 当前哪一层能跑、哪些 Node 正在跑”。
2. 执行如何装配上下文、挂接 tool / skill、解析输出并回写存储，由能力层负责；调度层只消费执行状态与结果事件，不展开实现细节。

## 5. 调度循环

```text
1) 从一级队列选 Topic（按 L0 -> L3，层内 RR）
2) 读取该 Topic 下所有未终态叶子 Node 与 TopicRuntimeState
3) 若 Topic 的 priority 索引为 dirty，则调用一次 PriorityAgent 重算 priority
4) 计算 active_priority
5) 找到 active_priority 对应的 runnable Node 集合
6) 若该集合为空，则将 Topic 标记为 stalled，并返回一级队列
7) 校验 current_node_id 是否仍属于 active layer；若不是，则按工程规则重选
8) 若 current_node_id 可派发且尚未运行，优先派发它
9) 使用剩余 Agent 容量继续派发 active layer 中其他未运行 Node
10) 每次派发成功都更新 running_node_ids
11) 能力层围绕该 Node 驱动执行流水线，并按 VFS 语义回写 session/output/progress/status/memory
12) 执行开始或产生首个有效进展时，更新 last_worked_node_id / last_worked_at
13) Node 完成、失败、阻塞、取消、恢复后，回收运行集并按需触发 priority 重算
14) 只有当前 active layer 清空或被显式重判后，才会开放下一优先级层
```

## 6. 并发与抢占

1. 并发能力首先受 Agent 池可用数限制，其次受 active layer 可执行 Node 数限制。
2. 同一 Topic 内允许同层并发，不允许跨层并发。
3. 同一 Node 同时最多只租给一个 Agent，避免重复写入和状态竞争。
4. 若有 5 个空闲 Agent，但当前 Topic 最高优先级层只有 2 个可执行 Node，那么这个 Topic 最多只会占用 2 个 Agent。
5. 低优先级层即使等待很久、即使系统还有闲置 Agent，也必须继续等待；这不是饿死，而是工作流阶段约束。
6. active layer 内的多个 Node 不需要互相抢占；它们可以直接并发执行。
7. 一级队列仍可在 Topic 之间做抢占：更高热度 Topic 进入后，可以回收低热度 Topic 的后续派发份额。
8. 二级层只有在 priority 重算后 active layer 发生变化时，才需要处理 Topic 内部的“层切换”。
9. 对于已经运行但跌出 active layer 的 Node，MVP 建议只做安全点回收或不再续租，不强求硬中断。
10. 因此，MVP 阶段的抢占重点仍然放在 Topic 级，而不是 Topic 内做复杂硬抢占。

## 7. 失败与重试

1. 可重试错误：对单个 Node 做指数退避 + 抖动重试。
2. Node 进入重试冷却后，应立即从 `running_node_ids` 移除。
3. 如果该 Node 仍属于 current active layer，则 Topic 仍然停留在这一层，不允许低优先级层越过它继续执行。
4. 不可重试错误：将 Node 标记为 `failed`，并按需触发 PriorityAgent 重算剩余 Node 的 priority。
5. 若 active layer 中部分 Node 失败但仍有其他同层 Node 可执行，则其他同层 Node 可以继续。
6. 若 active layer 中全部 Node 都处于 `blocked / retry_cooldown / waiting_external`，则 Topic 进入 stalled 状态，等待恢复、人工介入或优先级重判。
7. 超时执行：回收租约，更新 Node 状态，并根据策略进入 retry 或 failed。
8. 下游工具或模型熔断时，只冻结受影响 Node；但如果它们构成当前 active layer 的全部候选，则整个 Topic 会一起停住。
9. 二级层默认不使用“自动降级到下一层”来规避阻塞，否则会破坏阶段顺序。

## 9. 可观测性

核心指标建议包括：

1. `topic_queue_wait_ms`
2. `topic_node_queue_size`
3. `active_priority`
4. `active_priority_node_count`
5. `running_nodes_in_active_priority`
6. `priority_agent_invoke_count`
7. `priority_recompute_count`
8. `focus_pending_ms`
9. `current_node_stall_ms`
10. `current_node_switch_count`
11. `dispatch_latency_ms`
12. `preemption_count`
13. `affinity_hit_rate`
14. `throughput_node_executions_per_min`

这些指标的重点解释如下：

1. `focus_pending_ms` 应定义为 `current_node_id` 已选出但尚未进入 `running_node_ids` 的持续时间。
2. 在并发模型里，不建议再用 `current_node_id != last_worked_node_id` 直接计算该指标。
3. `current_node_stall_ms` 用于观测主焦点 Node 长时间无有效推进。
4. `active_priority_node_count` 与 `running_nodes_in_active_priority` 可以直接看出当前层是否被充分消费。
5. `priority_recompute_count` 过高通常意味着 Node 划分或阶段边界不稳定。

建议告警：

1. 任一高热度 Topic 等待时间超过阈值。
2. 当前 active layer 长时间 stalled。
3. `focus_pending_ms` 持续过高。
4. `switch_count` 在短窗口内异常升高。
5. PriorityAgent 调用频率异常升高。
6. retry rate 或 timeout rate 异常升高。
7. 高优先级层积压持续增长。

## 10. MVP 范围

1. 双队列基础实现。
2. 一级 Topic MLFQ。
3. 二级 Topic 内单逻辑队列 + Node priority 索引。
4. PriorityAgent 在入队与关键 dirty 事件上重算 Node priority。
5. `active_priority / current_node_id / running_node_ids / last_worked_node_id / last_worked_at / switch_count` 维护。
6. 同层并发、单 Node 单 Agent。
7. 严格禁止低优先级层偷跑。
8. 基础 retry / timeout / stalled 处理。
9. 简单 affinity 偏好，不做复杂学习型调度。
10. Topic 内部仅支持安全点回收，不做复杂硬抢占。

## 11. 并行开发分工建议

1. A 组：一级 MLFQ、TopicRuntimeState 持久化与二级 priority 索引。
2. B 组：PriorityAgent 接入、dirty 事件归并、active layer 计算与派发主循环。
3. C 组：Dispatch 租约、running_node_ids 生命周期、retry / timeout / stalled 处理。
4. D 组：指标、Tracing、告警与调度面板。
