# 模块契约

## 1. 当前决策

1. `vos` 使用 Go
2. `pool` 使用 Go 运行时，并保留 Python adapter
3. `schedule` 确认使用 Go
4. `agent` 继续保持 Python + Pydantic
5. 当前不引入 API、RPC、MCP，跨语言边界统一走 `CLI + JSON`

## 2. 模块技术栈与入口

### 2.1 VFS

1. 技术栈：Go
2. 当前命令入口：`cmd/vos`
3. 当前内部实现：`internal/vos/domain`、`internal/vos/store`、`internal/vos/service`、`internal/vos/cli`
4. 状态权责：`Topic / Node / tree / session / memory / input / output / progress / status`

### 2.2 Schedule

1. 技术栈：Go
2. 当前命令入口：`cmd/openmate-schedule`
3. 当前内部实现：`internal/schedule`
4. 状态权责：`TopicQueue / NodeQueue / TopicRuntimeState / dispatch / retry / stalled / preemption`

### 2.3 Pool

1. 技术栈：Go 运行时 + Python adapter
2. 当前命令入口：`cmd/openmate-pool`
3. 当前内部实现：`internal/poolgateway`
4. Python 适配层：`openmate_pool`
5. 状态权责：`invoke / attempt / usage / retry / circuit-break / capacity`

### 2.4 Agent

1. 技术栈：Python + Pydantic
2. 当前命令入口：`python -m openmate_agent`
3. 当前服务入口：`openmate_agent.service.AgentCapabilityService`
4. 状态权责：上下文注入、工具注入、skill 注入、执行装配

## 3. 调用边界规则

### 3.1 统一规则

1. 模块之间统一通过 CLI + JSON 通信
2. 该规则对 `Go -> Go`、`Go -> Python`、`Python -> Go` 一视同仁
3. `internal/*` 目录只服务模块自身实现与本模块测试，不作为跨模块依赖面
4. CLI 既是人工入口，也是模块通信入口

## 4. 具体契约

### 4.1 schedule <-> VFS

当前共识：

1. `schedule` 是 `Topic / Node` 的消费者，不是所有者
2. `schedule` 读写业务对象时，应通过 `vos` CLI + JSON 完成
3. `vos` CLI 既保留人工操作语义，也承担模块间调用语义

当前 VFS 已有能力：

1. Topic CRUD 基线
2. Node CRUD 基线
3. 叶子可操作性判断
4. 面向调度的叶子 Node 查询接口：
   - `vos node list --topic-id ... --leaf-only`
   - 可叠加 `--status` / `--exclude-status` 做状态过滤
5. 运行态字段原子更新约束：
   - `vos node update --expected-version N ...`
   - 基于 `Node.version` 做乐观并发校验
6. 子节点 `memory` 变更会聚合写入父节点 `memory` 缓存，供后续执行读路径复用

调度层当前实现：

1. `TopicRuntimeState` 已由 `schedule` 侧 SQLite 控制面持久化承载
2. 该结构仍保持在 schedule 控制面，不直接污染 VFS 的 `Topic / Node` 业务对象

### 4.2 schedule <-> pool

当前共识：

1. `schedule` 作为独立模块，应通过 `openmate-pool` CLI + JSON 消费 Pool 的标准化结果
2. `internal/poolgateway` 只服务 Pool 模块自身实现与本模块测试，不作为跨模块依赖面
3. Pool 对外稳定语义是：
   - `Invoke`
   - `Capacity`
   - `Records`
   - `Usage`
   - `Sync`

说明：

1. `Invoke` 是主执行入口
2. `Records` 与 `Usage` 属于观测和回看面，不替代执行主链路

### 4.3 agent <-> pool

当前共识：

1. Python Agent 通过 `openmate_pool.PoolGateway` 访问 Pool
2. `service.execute()` 只消费标准化 `InvokeResponse`
3. Python 侧不暴露底层 provider 调用细节

### 4.4 schedule <-> agent

当前状态：

1. `schedule` 已确认使用 Go
2. `agent` 继续保持 Python
3. 因此这一条边界必须定义为跨语言 worker 契约

当前已冻结项（MVP）：

1. worker 命令：`python -m openmate_agent.cli worker run --request-file <PATH>`
2. 输入/输出：`WorkerExecuteRequest/WorkerExecuteResponse`（JSON）
3. 退出码语义：
   - `0`：执行成功（`status=succeeded`）
   - `1`：执行失败（`status=failed`）
   - `2`：请求参数不合法
4. 调度原子：一次 worker 授权对应一次 `SessionEvent` 原子调用闭环
5. `priority_node` 与普通 Node 同构，必须经过 `schedule` 派发，不能旁路执行

当前仍待增强：

1. 取消 token 的执行中断语义仍需增强
2. retry/backoff 与 error taxonomy 仍需继续细化

### 4.5 统一 SQLite 数据库约定（一期）

当前决策：

1. `schedule runtime`、`pool gateway`、`vos session` 三类 SQLite 数据默认共用 `.openmate/runtime/openmate.db`。
2. `vos_state.json` 在一期保持独立，不并入 SQLite；默认路径为 `.openmate/runtime/vos_state.json`。
3. 统一入口参数约定：
   - `openmate-schedule --db-file`：默认同时作用于 `runtime-db-file` 与 `vos-session-db-file`。
   - `vos --db-file`：默认作用于 `session-db-file`。
   - `openmate-pool --db-file`：默认使用统一路径。
4. 显式覆盖参数优先级不变：当调用方显式设置 `--runtime-db-file` / `--vos-session-db-file` / `--session-db-file` 时，以显式值为准。
5. Python 适配层默认路径同步对齐：
   - `openmate_pool.PoolGateway`
   - `openmate_agent` 内的 `PoolGateway` 调用
   - `openmate_agent` 内的 `VosSessionGateway` / `VosContextGateway`
6. 一期不做历史数据库自动迁移或回填，旧库数据迁移由后续专项操作处理。

## 5. 主键与追踪约定

1. `topic_id`：项目边界主键
2. `node_id`：调度与执行的最小业务单元主键
3. `invocation_id`：一次标准化 LLM 调用主键
4. `attempt_id`：一次 provider 尝试主键
5. 当前不再额外引入独立 `task_id`；Node 自身就是“接着做”的对象
