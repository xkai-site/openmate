# AINative OS 架构总览

## 1. 目标

在应用层构建一个 AINative 操作系统：

1. 复用文件系统范式组织 `Topic` 与 `Node`。
2. 以 Agent 作为可调度的执行资源；若需要不同工作风格，优先在能力层通过 Prompt / Plan 组装，而不是在资源层固化角色。
3. 把调度、执行、能力注入、模型调用解耦，支持并行开发与演进。

## 2. 核心设计决策

1. 任务中心而非 CPU 中心：系统重点控制任务，不强绑定 Agent 与队列。
2. 双队列调度：一级按主题，二级按叶子 Node 优先级。
3. 交互驱动优先：一级优先级由用户交互热度决定，采用多级反馈思想。
4. 可插拔能力层：上下文、工具、skill、存储抽象化，避免过早绑定实现。

## 3. 模块拆分

1. 虚拟文件系统：`Node/Topic` 抽象、树结构组织、叶子节点可操作约束。
2. 调度队列：双队列、反馈调度、抢占与并发分配。
3. Agent 池：LLM 调用网关、执行治理、健康管理、调用级可观测采集。
4. Agent 能力：上下文注入、工具注入、skill 注入、对话存储。

## 4. 跨模块对接约定（2026-04-11）

### 4.1 Agent池对接面

1. 用户侧只维护 `model.json`。
2. 调度/能力模块向 Agent池提交标准化 LLM 请求，并从 Agent池接收标准化响应。
3. `service` 不再直接持有 provider 调用细节，只在 Agent池响应上做解析与业务映射。
4. 当前主调用契约为 `invoke`，查询与运维命令为 `cap / records / usage / sync`。
5. Agent池运行时已迁移为 Go；Python 侧只保留契约模型与 CLI adapter。
6. `model.json` 现已可配置重试参数，当前开放 `retry.max_attempts / retry.base_backoff_ms`。

### 4.2 状态与追踪约束

1. 当前主链路为：`node_id -> invocation_id -> api_id`。
2. 同一 invocation 下允许记录多次 attempt。
3. Agent池可在 invocation 内部对 `retryable` 的 attempt 做重试；上层消费最终响应，详细过程通过 `records` 回看。
4. 调用级 usage、latency、error、cost 由 Agent池统一采集。

### 4.3 并发与一致性约束

1. Agent池运行态使用 SQLite（WAL）。
2. Agent池内部以短事务执行预留与收尾步骤（`BEGIN IMMEDIATE`），避免并发超配。
3. 同时受 `max_concurrent`（API级）和 `global_max_concurrent`（全局级）约束。
4. 通道选择、调用执行、观测记录已在 Agent池内闭环完成。
