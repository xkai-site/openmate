# AINative OS 架构总览

## 0. 2026-04-14 补充

1. `pool` 已把主调用协议切到 OpenAI `Responses API`，不再以 `chat/completions` 作为主契约。
2. 上层模块与 `pool` 的对接形式仍为 `invoke / cap / records / usage / sync`，但 `invoke` 内层请求已改为 OpenAI `Responses create` 风格。
3. 工具调用相关字段如 `tools / tool_choice / previous_response_id / function_call_output` 由 `pool` 负责透传、重试与观测；工具执行本身仍留在上层。
4. `model.json.apis[*]` 新增 `request_defaults / headers / pricing`，由 `pool` 统一注入模型默认参数、附加请求头和成本定价。
5. usage 维度已升级为 `input_tokens / output_tokens / total_tokens / cached_input_tokens / reasoning_tokens`。

## 1. 目标

在应用层构建一个 AINative 操作系统：

1. 复用文件系统范式组织 `Topic` 与 `Node`。
2. 以调度层协调执行，而不是把 Agent、队列、资源直接绑死。
3. 把结构层、调度层、能力层、模型调用层解耦，支持并行开发与独立演进。

## 2. 当前模块与技术栈

1. 虚拟文件系统（VFS）：Go，实现 `Topic / Node` 结构与状态边界。
2. 调度队列（Schedule）：Go，负责控制平面、运行态与派发循环。
3. Agent 池（Pool）：Go 运行时 + Python adapter，负责标准化 LLM 调用网关。
4. Agent 能力（Agent）：Python + Pydantic，负责上下文、工具、skill 注入与执行装配。

## 3. 共享边界原则

1. `Go -> Go` 默认使用仓库内内部调用，不再强制走 CLI + JSON。
2. `Go -> Python` 与 `Python -> Go` 继续冻结在 `CLI + JSON` 边界。
3. 当前不引入 API、RPC、MCP；CLI 同时承担人工入口、测试入口与兼容入口。
4. 每一层只拥有自己的状态权责，不跨层偷写别人的运行态。

## 4. 当前冻结的跨模块契约

### 4.1 VFS

1. VFS 是 `Topic / Node` 的唯一权威状态层。
2. 当前 Go 代码位于 `internal/vos/*`，对外命令入口是 `cmd/vos`。
3. 同进程 Go 场景下，`schedule/httpapi` 默认通过 `internal/vos/service` 直调。
4. `vos` CLI 保留人工与兼容入口语义。

### 4.2 Pool

1. Pool 是 LLM 调用、并发治理、重试、熔断与调用级观测的唯一权威层。
2. 当前 Go 代码位于 `internal/poolgateway`，对外命令入口是 `cmd/openmate-pool`。
3. Pool 已纳入统一 Go 运行时装配层，与 VFS/Schedule 共用运行时配置。
4. Python 调用方继续通过 `openmate_pool` adapter 访问；该 adapter 本质上仍是对 Go CLI 的封装。

### 4.3 Agent

1. Agent 能力层继续保持 Python + Pydantic。
2. 当前主要稳定边界是 `openmate_agent.service` 与 `tool` CLI。
3. 能力层负责提示词装配、工具注入、skill 注入，不负责调度决策与 provider 调用。
4. 已新增稳定 worker 入口：`python -m openmate_agent.cli worker run --request-file <PATH>`。

### 4.4 Schedule

1. Schedule 现确认采用 Go 技术栈推进。
2. Schedule 是控制平面，负责 `TopicQueue / NodeQueue / TopicRuntimeState / dispatch`。
3. Schedule 不直接实现 Prompt 编排，不直接实现 provider 调用。
4. Schedule 对接 VFS 时默认走内部调用；对接 Agent 继续走 worker CLI + JSON。

## 5. 主链路与标识

1. `topic_id`：项目边界主键。
2. `node_id`：调度与执行的最小业务单元主键。
3. `invocation_id`：一次标准化 LLM 调用的主键。
4. `attempt_id`：同一 invocation 下单次 provider 尝试的主键。
5. Schedule 运行态字段如 `current_node_id / active_priority / running_node_ids` 属于控制面状态，不替代业务主键。

## 6. 当前待补缺口

1. `priority_node` 当前为启发式重排，尚未接入真实 PriorityAgent 策略与提示词链路。
2. 一级 Topic 热度输入仍需接入真实交互事件流，MLFQ 提级/降级规则需继续参数化。
3. worker 的取消 token / 超时回收 / 重试分类仍需继续增强。
4. 部分 Windows 环境对系统默认 Go cache 目录无写权限，测试与构建需显式落到仓库内 `.openmate/`。
