# 调度队列架构

## 1. 模块目标

调度层负责把“待执行工作”公平且高效地分配给 Agent 执行资源，兼顾用户交互响应、吞吐与成本。

核心目标：

1. 一级按主题调度，解决“用户当前最关心什么”。
2. 二级按主题内工作流调度，解决“同主题内先做什么”。
3. 与 Agent 解耦，支持动态并发与抢占。
4. 可解释调度决策，便于调优和排障。

## 2. 设计结论

采用双队列：

1. 一级队列（TopicQueue）：主题级多级反馈队列（MLFQ）。
2. 二级队列（NodeQueue）：主题内部的可执行叶子 Node 优先队列。

调度原则：

1. 调度与执行分离。
2. Agent 是执行槽位，不与队列强绑定。
3. 一级队列使用操作系统最纯粹的MLFQ机制，管理的是Topic。
4. 二级队列由 Topic 内叶子 Node 的状态与工作流关系共同决定，解决同主题任务下先后顺序。
5. Agent affinity 只作为派发优化项，不作为队列切分依据。
6. 同级轮转防止饿死，时间片耗尽降级，周期性提级。

## 3. 一级队列（主题级）

### 3.1 队列分层

建议四级：

1. `L0`：最高优先，强交互主题。
2. `L1`：活跃主题。
3. `L2`：一般主题。
4. `L3`：低频主题。

### 3.2 进入与迁移规则

1. 新激活主题默认进入 `L0`。
2. 主题用完时间片且仍有待执行任务，降一级。
3. 定时全局提级，避免长期饥饿。
4. 发生新交互事件时按热度可直接提至最高级。

### 3.3 热度计算（建议）

这里的 `score` 只回答一个问题：用户当前注意力主要在哪个 Topic 上。

它不是业务价值评分，也不是让系统猜“哪个任务更重要”。

MVP 阶段建议先采用容易解释的规则：

1. 用户当前正在交互的 Topic 直接提升到高优先级。
2. 最近有用户消息、手动操作的 Topic 提级。
3. 长时间无交互的 Topic 按衰减规则降级。
4. 同等热度下，最近已经拿到较多执行份额的 Topic 可加轻微惩罚，避免独占。

如果后续需要量化实现，再落成启发式分数：

```text
score = interaction_heat
      - idle_decay
      - recent_service_penalty
```

其中：

1. `interaction_heat` 由 `recent_user_messages / recent_manual_actions / explicit_focus_flag` 构成。
2. `idle_decay` 用于让长时间无交互主题自然回落。
3. `recent_service_penalty` 用于抑制热点主题长期占满全部 Agent。

最终按阈值映射到 `L0~L3`。

## 4. 二级队列（主题内工作流）

二级队列位于每个 Topic 内部，它维护的是该 Topic 中可执行叶子 Node 的执行优先级。

它的职责是：当某个 Topic 已经从一级队列获得执行份额后，从该 Topic 的叶子 Node 中挑出最值得先执行的一批 Node。这里的队列是优先队列，优先级计划由人/Agent确定，可以边做边调整。

换句话说：

1. 一级队列解决“现在哪个 Topic 该获得系统服务”。
2. 二级队列解决“这个 Topic 内下一步在哪个 Node 上工作”。
3. Agent 不拥有自己的专属任务队列，而是被派发到具体 Node 上工作。

### 4.1 核心语义

1. 二级队列归属于 `Topic`，而不是归属于 `Agent`；每个 Topic 独立维护自己的局部工作队列。
2. 二级队列的主对象是同一 Topic 下所有“未终态”的叶子 `Node`，因为叶子 Node 才是用户直接操作、也是 Agent 直接工作的对象。
3. 二级层采用 continuation-first 策略：默认优先继续当前焦点 Node，而不是每次重新审视整个队列。
4. `Topic` 需要额外维护一个 `current_node_id`，表示“当前默认应该接着做谁”。
5. 只有当 `current_node_id` 不存在、当前节点不能继续、或用户/系统显式要求切换时，才引入优先级 Agent 做一次切换裁决。
6. `Node` 是长期状态与产物的承载体，负责统一沉淀 `session / input / output / progress / status / memory`。
7. 调度层不再引入额外任务对象；Node 自身就是“接着做”的对象。
8. 同一个 Node 在生命周期中可以被多次调度；每次执行都继续在该 Node 上追加状态、对话和产物。
9. 同一 Node 默认不应被重复并发派发，除非该 Node 明确声明可重入。

### 4.2 队列与焦点字段

二级队列本质上是 Topic 内的单队列，但它不是“每轮都做全量重排”的传统优先队列，而是“队列 + 当前焦点”的组合。

建议至少维护以下运行时字段：

```ts
interface TopicRuntimeState {
  topicId: string;
  current_node_id?: string;
  last_worked_node_id?: string;
  last_worked_at?: string;
  switch_count: number;
}
```

其中：

1. 队列元素是同一 Topic 下所有未终态的叶子 `Node`。
2. `current_node_id` 表示当前焦点 Node，也就是默认续做对象。
3. `last_worked_node_id` 表示最近一次被 Agent 实际处理过的 Node。
4. `last_worked_at` 表示最近一次实际推进该 Topic 的时间。
5. `switch_count` 表示该 Topic 在统计窗口内的焦点切换次数。

这些字段的职责边界应当非常明确：

1. 队列保存“这个 Topic 里还有哪些 Node 值得继续管理”。
2. `current_node_id` 回答“现在默认继续谁”。
3. `last_worked_*` 和 `switch_count` 只属于工程侧调度历史，不属于 Node 的业务语义。

### 4.3 调度决策（建议）

每次调度某个 Topic 时，不先全量重排，而是先判断当前焦点是否还能继续。

```text
if current_node_id exists and can_continue(current_node_id):
  dispatch(current_node_id)
else:
  next_node_id = priority_agent.choose(queue_snapshots)
  current_node_id = next_node_id
  dispatch(current_node_id)
```

这里的 `can_continue(node)` 建议至少检查：

1. Node 仍然存在，且仍是叶子 Node。
2. Node 尚未进入 `done / failed / cancelled` 终态。
3. Node 当前不处于显式阻塞状态。
4. Node 不在重试冷却窗口内。
5. 用户没有显式要求切换焦点。
6. 当前 Node 没有连续多轮“无进展”到需要重新裁决。

这套策略的核心不是“永远不切换”，而是：

1. 能继续就继续。
2. 只有在继续不合理时，才为切换支付一次判断成本。

### 4.4 工程侧 AOP 维护

`current_node_id`、`last_worked_node_id`、`last_worked_at`、`switch_count` 不应由 Agent 显式维护，而应由系统在关键动作上自动更新。

这里的 AOP 可以理解为横切的工程记录逻辑，例如：

1. 派发成功时，自动更新 `last_worked_node_id` 与 `last_worked_at`。
2. `current_node_id` 从 A 切到 B 时，自动执行 `switch_count += 1`。
3. 当前焦点 Node 完成、阻塞、进入重试或被取消时，自动清空或失效化 `current_node_id`。
4. 当前焦点 Node 回写了新进展时，自动刷新该 Topic 的最近推进时间。

第三项字段的意义分别是：

1. `last_worked_node_id`
作用：记录刚才实际工作过的是谁，给“接着做”提供最近现场。

2. `last_worked_at`
作用：记录这个 Topic 最近一次真实推进时间，用于判断是否卡住、是否长期无进展。

3. `switch_count`
作用：记录该 Topic 焦点切换频率，用于观测是否在来回跳任务，也可作为后续抑制过度切换的依据。

这些信息的价值在于：

1. 不需要 Agent 记住“刚才在做什么”，系统自己有调度历史。
2. 不需要把调度历史污染到 Node 业务对象中。
3. 优先级 Agent 在做切换裁决时，可以获得更稳定的上下文事实。

### 4.5 优先级 Agent 的职责

优先级 Agent 不是常驻调度器，而是“焦点失效时的一次性仲裁器”。

它只在以下场景接入：

1. `current_node_id` 为空。
2. 当前 Node 已完成。
3. 当前 Node 进入阻塞或重试冷却。
4. 当前 Node 连续多轮无有效进展。
5. 用户明确要求切换方向。

优先级 Agent 读取的是队列里每个 Node 的轻量快照，而不是完整执行上下文。

建议输入至少包含：

1. `nodeId`
2. `name`
3. `description`
4. `status`
5. `progress`
6. `last_worked_at`
7. `isCurrent`

优先级 Agent 的输出应尽量收窄：

1. 正常情况下只返回一个 `next_node_id`
2. 当没有合适目标时，允许返回空结果

系统接到结果后的动作也应收窄：

1. 将 `current_node_id` 切到 `next_node_id`
2. 必要时把该 Node 提升到队首
3. 不要求优先级 Agent 返回完整排序结果

这样做的目的是：

1. 把“谁最值得接着做”交给 Agent 判断。
2. 但不把“整个队列如何维护”彻底交给 Agent 接管。

### 4.6 Agent 在 Node 上的工作方式

Agent 一旦被派发到某个 Node，就直接围绕该 Node 的结构工作，而不是再额外维护一套平行对象。

建议最少复用以下路径：

1. 读 `Topic.meta_ref`，获得项目级必读信息。
2. 读当前 `Node.session`，作为当前对话上下文。
3. 读当前 `Node.input / output / progress / status`，获得执行现场。
4. 读父节点 `memory`，获得共享记忆层。
5. 执行完成后回写当前 `Node.session / output / progress / status`。
6. 若形成新的长期记忆，再更新当前 Node `memory`，并按 VFS 规则触发父节点记忆聚合。

这样设计的意义是：

1. Agent 直接在 Node 上工作，用户和系统看到的是同一个对象。
2. 输出、对话、进展、记忆都沉淀在 Node 体系内，不需要额外复制一套任务结果结构。
3. 调度层只负责“把哪个 Agent 派到哪个 Node 上”，而不是接管 Node 的业务语义。

## 5. 调度循环

```text
1) 从一级队列选主题（按 L0 -> L3，层内 RR）
2) 读取该 Topic 的 `current_node_id`
3) 判断当前焦点 Node 是否还能继续
4) 若能继续，则直接向 Agent 池申请执行该 Node
5) 若不能继续，则调用优先级 Agent 从队列轻量快照中选出 `next_node_id`
6) 更新 `current_node_id`，必要时提升该 Node 到队首
7) Agent 在 Node 上执行，直接读写 Node 的 session/output/progress/status/memory
8) 回收结果，更新 Topic 热度、Node 状态和 AOP 运行时字段
9) 执行一级升降级与必要的二级焦点切换
```

## 6. 并发与抢占

1. 系统并发上限由 Agent 池实时可用数决定。
2. continuation-first 意味着同一 Topic 默认只维护一个 `current_node_id` 作为主焦点。
3. 因此同一 Topic 在 MVP 阶段默认只推进一个 Node；富余并发优先让给其他 Topic。
4. 如果后续需要同 Topic 多 Agent 并发，建议优先在当前 Node 的执行计划内部解决，而不是让二级队列同时推进多个焦点 Node。
5. Agent affinity 只用于“当前焦点 Node 与 Agent”的匹配优化，不应反向污染焦点选择。
6. 抢占触发条件之一是更高优先级主题进入且等待超阈值。
7. 抢占本质上是 Topic 级抢占：优先回收低优先 Topic 的当前执行槽位。
8. 抢占只发生在可安全中断或支持检查点恢复的执行上。
9. Topic 内部从 A Node 切到 B Node，不视为系统抢占，而视为本地焦点切换。

## 7. 失败与重试

1. 可重试错误：退避重试（指数退避 + 抖动）。
2. 若当前焦点 Node 进入重试冷却，应清空或失效化 `current_node_id`，等待下一轮重新裁决。
3. 不可重试错误：标记失败并回写 Node 状态；若该 Node 是当前焦点，则移除焦点。
4. 超时执行：回收租约，更新 Node 状态，并根据策略进入重试或失败封顶。
5. 熔断：下游模型或工具异常时暂缓派发相关 Node；若当前焦点受影响，则触发切换裁决。

## 8. 关键接口

### 8.1 输入事件

1. `LeafNodeCreated`
2. `NodeStatusChanged`
3. `NodeProgressUpdated`
4. `NodeRetryDue`
5. `NodePriorityHintUpdated`
6. `TopicInteractionUpdated`
7. `NodeCancelled`
8. `AgentCapacityChanged`

### 8.2 输出事件

1. `DispatchRequested`
2. `DispatchLeaseExpired`
3. `PriorityAgentRequested`
4. `CurrentNodeSwitched`
5. `NodeExecutionFinished`
6. `NodeExecutionDeadLettered`

## 9. 可观测性

核心指标：

1. `topic_queue_wait_ms`（分 Topic/L0~L3）
2. `topic_node_queue_size`
3. `continuation_hit_rate`
4. `priority_agent_invoke_count`
5. `current_node_stall_ms`
6. `current_node_switch_count`
7. `dispatch_latency_ms`
8. `preemption_count`
9. `affinity_hit_rate`
10. `throughput_node_executions_per_min`

建议告警：

1. 任一主题等待时间超过阈值
2. `current_node` 长时间无进展
3. `switch_count` 在短窗口内异常升高
4. 优先级 Agent 调用频率异常升高
5. 重试率异常升高
6. 高优先主题积压持续增长

## 10. MVP 范围

1. 双队列基础实现（无复杂抢占）
2. 一级 MLFQ + 二级单队列
3. `current_node_id` 与 AOP 运行时字段维护
4. continuation-first 调度
5. 优先级 Agent 仅返回 `next_node_id`
6. 基础重试与超时
7. 简单 affinity 偏好，不做复杂亲和学习
8. 不展开同 Topic 多焦点并发

## 11. 并行开发分工建议

1. A 组：一级队列内核 + Topic 二级单队列 + `current_node_id` 状态维护。
2. B 组：continuation-first 主循环、优先级 Agent 接入与票据管理。
3. C 组：AOP 埋点、指标、追踪、告警与调参面板。
